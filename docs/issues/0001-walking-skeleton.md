# Issue 0001: Walking skeleton - select an element, change font-size, save a patch

> Labels: enhancement, ready-for-agent · Type: AFK

## Parent

[PRD: webtweak v1](../PRD.md)

## What to build

The complete thin path through every layer, end to end. Running `webtweak <page.html>` boots the local server, serves the page from its own directory, and injects the Overlay before `</body>`. In the browser, hovering highlights the element under the cursor and clicking selects it, showing a selection box and a structure breadcrumb. The properties panel exposes a single control - font-size - which previews live on the selected element. Clicking Save POSTs the session to the server, which writes a pending Batch containing one fingerprinted Patch to `<page-stem>.webtweak.json` next to the source. Source is never mutated.

This slice establishes: the CLI + server, the `inject_overlay` pure function, the `apply_batch` pure function (extracted from the existing inline save logic, which already implements the full running-history behaviour), the Overlay scaffolding (bar, panel, hover/selection boxes, breadcrumb), the element picker, the `fingerprint`/`cssPath` capture, the single property-control pattern, and the save round-trip.

## Agent Brief

**Category:** enhancement
**Summary:** Ship the walking-skeleton tracer bullet - select an element, change its font-size, and save a fingerprinted Patch to the edits file - extracting the two deep modules as pure, tested functions along the way.

**Current behavior:**
The `webtweak` server already serves a local page with the Overlay injected, serves Overlay assets under a reserved path, and handles a save POST. Both the HTML-injection logic and the save logic currently live inline in the request handler; the save logic already implements the full running-history behaviour (overwrite the same-session pending Batch vs append, with a viewport + timestamp stamp and a `pending` status). There is no browser-side Overlay yet - nothing can be selected or edited, and there is no element picker, properties panel, fingerprint capture, or save trigger. No automated tests and no sample fixture page exist.

**Desired behavior:**
Running `webtweak <page.html>` opens the page with a working Overlay. Hovering highlights the element under the cursor without ever highlighting the Overlay's own UI. Clicking selects an element, showing a selection box and an ancestry breadcrumb. A properties panel offers a single font-size control seeded from the element's computed value; changing it previews live on the element. Clicking Save sends the session to the server, which writes a pending Batch holding one Patch to `<page-stem>.webtweak.json` beside the source. The page's source file is never modified by a session.

**Key interfaces:**
- `inject_overlay(html, target_name) -> html` - extract the existing inline injection into a pure module-level function: insert the Overlay markup immediately before the final `</body>` (append if none present), carrying `target_name` in the injected config object. No HTTP dependency.
- `apply_batch(doc, payload, now) -> doc` - extract the existing save logic into a pure module-level function and have the request handler delegate to it. Preserve current behaviour: locate the payload's same-session `pending` Batch and replace it, else append a new pending Batch; stamp viewport and save-time; default status `pending`. Do not reduce it to append-only.
- Overlay `fingerprint(el)` - returns the identity bundle: tag, id, classes (excluding `wt-*`), truncated text, a best-effort `cssPath` selector, and a clean opening tag with any injected inline `style` stripped.
- Overlay `cssPath(el)` - an `nth-of-type` selector path stopping at the nearest id or `body`.
- Save payload shape - `{ sessionId, viewport, patches: [ { fingerprint, changes } ] }` POSTed to the reserved save endpoint; `changes` for this slice carries only `font-size`.

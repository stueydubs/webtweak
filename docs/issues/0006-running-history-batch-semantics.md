This issue completes the test coverage for the remaining and edge cases, and corrects the behaviour if any test exposes a gap.

## Acceptance criteria

- [ ] Unit test: Batches with status `reconciled` are never modified by a save (a new pending Batch is appended instead of overwriting a reconciled one, even for the same `sessionId`)

- [ ] Unit test: an empty/new edits document is initialised correctly on first save
- [ ] Unit test: an existing document with a mix of `reconciled` and `pending` Batches preserves order and only the matching pending Batch is replaced
- [ ] Any behaviour gap surfaced by the above is fixed in `apply_batch`

## Agent Brief

**Category:** enhancement
**Summary:** Complete unit-test coverage for the running-history semantics of `apply_batch`, including the cases the walking skeleton's tests do not require.

**Current behavior:**
`apply_batch` already implements full running-history semantics and ships as a pure function in 0001 with tests for the core cases (first save creates a pending Batch, same-session re-save overwrites, new session appends, viewport/save-time stamped). The remaining edge cases are unproven.

**Desired behavior:**
`apply_batch` has comprehensive test coverage. In particular it is proven that `reconciled` Batches are immune to saves, that missing `sessionId` is handled deterministically, and that batch ordering is preserved when a document mixes `reconciled` and `pending` Batches. Any behaviour gap a test exposes is fixed in `apply_batch` itself.

**Key interfaces:**
- `apply_batch(doc, payload, now) -> doc` - the pure function extracted in 0001. Tests exercise it directly through its public signature, never via HTTP or private helpers.

**Acceptance criteria:**
- [ ] Test: a `reconciled` Batch is never modified by a save - even a same-`sessionId` save appends a new pending Batch rather than overwriting it
- [ ] Test: a save with no `sessionId` is handled deterministically and does not corrupt or merge unrelated Batches
- [ ] Test: an empty/new edits document is initialised correctly on first save
- [ ] Test: a document mixing `reconciled` and `pending` Batches preserves order; only the matching pending Batch is replaced
- [ ] Any behaviour gap surfaced is fixed in `apply_batch`

**Out of scope:**
- Changing the Batch schema or the `pending`/`reconciled` status model
- The reconcile step that sets Batches to `reconciled` (done by Claude, not this codebase)
